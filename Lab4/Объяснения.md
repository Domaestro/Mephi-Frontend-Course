## Часть 1. SCSS / SASS / Less

### Зачем нужны препроцессоры

Чистый CSS не умеет:
- хранить переменные,
- вкладывать селекторы друг в друга,
- использовать циклы,
- переиспользовать блоки правил (миксины).

Препроцессор — это программа, которая принимает на вход файл `.scss` /
`.sass` / `.less` и превращает его в обычный `.css`, который понимает
браузер.

### Чем отличаются SCSS, SASS и Less

- **SCSS** — синтаксис, очень похожий на CSS. Используются `{ }` и `;`.
  Любой валидный CSS — это валидный SCSS.
- **SASS** — старший «индентационный» синтаксис того же самого
  компилятора. Вместо скобок и точек с запятой — отступы и переносы
  строк (как в Python).
- **Less** — отдельный препроцессор от других авторов. Похож на SCSS,
  но переменные начинаются с `@`, миксины объявляются по-другому.

В лабораторной мы реализуем одно и то же сначала на SCSS (задания 2-3),
потом переписываем на SASS (задание 4) и на Less (задание 5).

В `Lab4/scss/package.json` объявлены зависимости:

```json
"devDependencies": {
    "less": "^4.2.0",
    "sass": "^1.77.0"

```

После `npm install` появится локальный `node_modules` с обеими

Скрипт `watch` запускает компилятор `sass` в режиме отслеживания

времени» из условия задания. Документация:
<https://sass-scss.ru/documentation/cli/dart-sass/> (раздел «watch»).

### Задание 2 — список из 100 чисел

HTML-страница простая: пустой `<ul>` и небольшой скрипт, который
создаёт 100 элементов `<li class="num-N">` с цифрой N. Раскрашивание —
полностью на совести SCSS:

```scss
@for $n from 1 through 100 {
    .num-#{$n} {
        color: rgb($n, 256 - $n, 0);
    }
}
```

- `@for $n from 1 through 100` — стандартный SCSS-цикл.
  Конструкция взята из официальной документации
  <https://sass-scss.ru/documentation/at-rules/control/for/>. Слово
  `through` (а не `to`) включает правую границу.
- `#{$n}` — интерполяция: значение переменной подставляется прямо в
  имя селектора. Без интерполяции мы получили бы класс `.num-$n`
  буквально.
- `rgb($n, 256 - $n, 0)` — арифметика на лету. `$n` пробегает 1…100,
  поэтому красный растёт, зелёный убывает, как и просили в условии.

После компиляции в `styles.css` оказывается 100 правил вида
`.num-1 { color: rgb(1, 255, 0); }`.

### Задание 3 — макет страницы

Используются три ключевые вещи: переменные, миксины, вложенные правила.

#### Переменные

```scss
$panel-color: #e8e8e8;
$panel-padding: 18px 24px;
$page-width: 820px;
```

Зачем: цвет панели встречается несколько раз. Если положить его в
переменную, поменять оформление можно в одном месте. Описание
переменных в SCSS: <https://sass-scss.ru/documentation/variables/>.

#### Миксины

```scss
@mixin panel {
    background-color: $panel-color;
    padding: $panel-padding;
    border-radius: $panel-radius;
    margin-bottom: $gap;
}
```

`@mixin` — это «именованный кусок CSS». Подключается через `@include`:

```scss
.menu {
    @include panel;
    ...
}
```

В условии задания миксины обязаны быть, поэтому общий вид панели
(серый фон, скругление, отступ снизу) вынесен именно в миксин `panel`,
а отдельно `centered` — для центрирования текста. Документация:
<https://sass-scss.ru/documentation/at-rules/mixin/>.

#### Вложенные правила

```scss
.menu {
    @include panel;
    ul {
        list-style: none;
        ...
        li a {
            color: $link-color;
        }
    }
}
```

Это второй обязательный пункт условия. Вложенность даёт код,
повторяющий структуру HTML, и компилируется в обычные селекторы:
`.menu ul`, `.menu ul li a`. Документация:
<https://sass-scss.ru/documentation/style-rules/>.

### Задание 4 — синтаксис SASS

Тот же код, но без `{ }` и без `;`. Иерархия задаётся отступами:

```sass
.menu
    +panel

    ul
        list-style: none
        ...
        li a
            color: $link-color
```

Тонкости:
- `+panel` — короткая запись `@include panel`. Допустим и
  длинный вариант, я выбрал `+`, потому что он короче и часто
  встречается в реальных проектах SASS.
- Циклы и `@mixin` объявляются точно так же, без скобок.
- Файл должен называться с расширением `.sass`, иначе компилятор
  ожидает фигурные скобки.

### Задание 5 — синтаксис Less

Главные отличия от SCSS:
- Переменные начинаются с `@`, а не с `$`: `@panel-color`.
- Миксин — это просто класс с пустыми скобками: `.panel() { … }`.
  Подключается как `.panel();` внутри другого правила.
- В Less нет `@for`. Вместо него используется **рекурсивный миксин с
  условием** (так называемый «mixin guard»):

```less
.colorize(@n) when (@n > 0) {
    .colorize(@n - 1);

    .num-@{n} {
        color: rgb(@n, 256 - @n, 0);
    }
}

.colorize(100);
```

Эта конструкция — официальный приём из документации Less:
<https://lesscss.ru/features/#guards-feature>. Сначала миксин зовёт
сам себя для `n - 1`, потом печатает текущее правило, поэтому в
итоге получаются классы `.num-1 … .num-100`.

## Часть 2. TypeScript

### Что такое TypeScript

> «TypeScript — язык программирования, надмножество JavaScript … TS
> проверяет типы во время компиляции и выдаёт ошибку, если переменной
> когда-либо будет присвоено значение другого типа.»
> *(TypeScript+1.pdf, слайд «TypeScript - what is it»)*

То есть мы пишем код почти как на JS, но добавляем аннотации типов.
Компилятор `tsc` проверяет всё это и превращает в обычный JavaScript.

### Установка и watch (задание 1)

В `Lab4/typescript/package.json` объявлен `typescript` как локальная
зависимость. В `tsconfig.json` указаны:

- `target: ES2020` — версия итогового JS;
- `module: commonjs` — формат модулей (нужен, чтобы Node.js понял
  скомпилированный код);
- `strict: true`;
- `noImplicitAny: true` (на случай, если кто-то выключит часть
  `strict`);
- `rootDir: ./src`, `outDir: ./dist` — откуда брать `.ts` и куда
  складывать `.js`.

Описание `tsconfig.json` есть в лекции *(TypeScript+1.pdf, слайд
«Config»)*. Команды watch / однократная сборка — `tsc --watch` и
`tsc` соответственно *(тот же PDF, слайды «First steps» и
«Strictness»)*.

### Задание 2 — класс `User` + интерфейс

```ts
interface IUser {
    name: string;
    age: number;
    hello(): void;
}

class User implements IUser {
    name: string;
    age: number;
    constructor(name: string, age: number) { ... }
    hello(): void { ... }
}
```

- **Интерфейс** — это описание формы объекта. См. слайд *«Type aliases
  & interfaces»* (TypeScript+1.pdf): «Если тип используется в
  нескольких местах, его можно переиспользовать. Для этих целей служат
  синонимы типов и интерфейсы».
- **`implements`** — TS проверяет, что класс действительно соответствует
  интерфейсу. Если убрать любой метод или поле, компилятор сразу
  ругнётся. См. слайд *«Implements»* (TypeScript+2.pdf):
  «Ключевое слово implements используется для проверки, соответствует
  ли класс определенному интерфейсу».
- Возвращаемый тип `void` для `hello()` берём с слайда *«Functions»*
  (TypeScript+1.pdf): «И какое значение возвращает».
- Сообщение собирается через шаблонный литерал
  ``` `Hi! My name is ${this.name}…` ``` — стандартный JS, упомянут на
  слайде *«Downleveling»* (TypeScript+1.pdf).

### Задание 3 — псевдоним типа

Тот же класс, но описание формы — через `type`:

```ts
type TUser = {
    name: string;
    age: number;
    hello(): void;
};
class UserAlias implements TUser { ... }
```

Лекция (слайд *«Type aliases & interfaces»*, TypeScript+1.pdf) явно
показывает, что `type` и `interface` обычно взаимозаменяемы. Главное
отличие, которое стоит запомнить (слайд *«Type vs Interface»*): `type`
нельзя расширить повторным объявлением, для расширения используется
пересечение `&`. Здесь нам этого не нужно.

### Задание 4 — перегруженная `distance`

```ts
type Point = { x: number; y: number };

function distance(x1: number, y1: number, x2: number, y2: number): number;
function distance(p1: Point, p2: Point): number;
function distance(
    arg1: number | Point,
    arg2: number | Point,
    arg3?: number,
    arg4?: number
): number {
    ...
}
```

- Две первые строки — **сигнатуры перегрузки** (overload signatures).
  Третья — **сигнатура реализации** (implementation signature).
  Терминология взята со слайда *«Function overload»* (TypeScript+1.pdf):
  «Сигнатура реализации не учитывается при вызове и служит только для
  определения тела. Сигнатура реализации должна быть совместима с
  сигнатурами перегрузок».
- Тип `Point` — псевдоним для объекта с полями `x` и `y` (см. слайд
  *«object»*, TypeScript+1.pdf).
- В реализации делаются проверки `typeof arg1 === 'number'` —
  стандартный приём «сужение типа», описанный на слайде *«typeof
  operator»* (TypeScript+1.pdf).
- `any` намеренно не используется (запрет в условии, а также в
  лекции на слайде *«any»*: «Злоупотреблять им не стоит»).
- `?` после имени параметра делает его необязательным — слайд
  *«Optional parameters»* (TypeScript+1.pdf).

### Задание 5 — бинарное дерево

#### Главная сложность задания

Условие требует:
- настройки `strict` и `noImplicitAny` (включены в `tsconfig.json`);
- **избегать утверждения типов** — это операторы `as` и `!`. Слайды
  *«Type assertion»* и *«null & undefined»* (TypeScript+1.pdf) явно
  предупреждают: `as` сужает тип «руками», `!` отбрасывает `null`
  без проверки. Оба обхода типобезопасности нам запрещены.

Поэтому везде, где значение может быть `null`, я ставлю **обычные
проверки `=== null`**:

```ts
private insertNode(node: TreeNode<T> | null, value: T): TreeNode<T> {
    if (node === null) {
        return new TreeNode(value);
    }
    ...
}
```

После такой проверки TS сам понимает, что в `else`-ветке `node`
точно не `null` — это та же «control flow analysis» со слайда
*«Control flow analysis»* (TypeScript+1.pdf).

#### Дженерики

Дерево хранит значения произвольного типа `T`:

```ts
class TreeNode<T> { ... }
class BinaryTree<T> { ... }
```

`<T>` — параметр типа, по аналогии с лекцией *«Generic functions»* и
*«Generic classes»* (TypeScript+1.pdf и TypeScript+2.pdf). Это
позволяет создавать `BinaryTree<number>`, `BinaryTree<string>` и т. п.,
не дублируя код.

#### Компаратор

Я не предполагаю, что значения умеют сравниваться через `<` и `>`
(строки тоже могли бы быть в дереве). Поэтому передаю функцию
сравнения:

```ts
type Comparator<T> = (a: T, b: T) => number;

constructor(compare: Comparator<T>) {
    this.compare = compare;
}
```

Тип `(a: T, b: T) => number` — это «выражение типа функции» со слайда
*«Function type expressions»* (TypeScript+1.pdf).

#### Методы

- `insert`, `search`, `remove`, `update`, `height` — публичные методы;
- `insertNode`, `searchNode`, `removeNode`, `findMin`, `heightOf` —
  закрытые рекурсивные помощники с модификатором `private` (слайд
  *«Class members visibility»*, TypeScript+2.pdf).
- `update(oldValue, newValue)` сделан как «удалить старое, вставить
  новое». Просто переписывать `value` в найденном узле нельзя — нарушится
  свойство BST (левый < узел < правый), и поиск перестанет работать.

#### Высота

Стандартный рекурсивный подход:
```ts
private heightOf(node: TreeNode<T> | null): number {
    if (node === null) return 0;
    return 1 + Math.max(this.heightOf(node.left), this.heightOf(node.right));
}
```

### Задание 6 — паттерны проектирования

#### Adapter (структурный)

> «Adapter — структурный паттерн, позволяющий классам с
> несовместимыми сигнатурами методов взаимодействовать между собой».
> *(TypeScript+2.pdf, слайд «Adapter»)*

Пример: датчик `CelsiusSensor` отдаёт градусы Цельсия. Внешний код
работает с интерфейсом `FahrenheitSensor` (метод `readF()`). Адаптер
оборачивает датчик и переводит результат:

```ts
class CelsiusToFahrenheitAdapter implements FahrenheitSensor {
    constructor(private readonly source: CelsiusSensor) {}
    readF(): number { return this.source.read() * 9 / 5 + 32; }
}
```

Структура повторяет пример из лекции (`XmlToPdfAdapter` на следующем
слайде той же темы): один класс хранит ссылку на адаптируемый объект и
имеет методы целевого интерфейса.

#### Strategy (поведенческий)

> «Strategy — поведенческий паттерн, который определяет схожие
> алгоритмы и помещает их в разные классы, после чего эти алгоритмы
> можно взаимозаменять во время исполнения программы».
> *(TypeScript+2.pdf, слайд «Strategy»)*

У нас интерфейс `PriceStrategy` и три реализации: `RegularPrice`,
`HolidayDiscount`, `VipDiscount`. Класс `Cart` хранит ссылку на
стратегию и в любой момент позволяет её заменить методом
`setStrategy`. Контракт, имена `setStrategy` / `doSomeBusinessLogic`
из лекции сохранены, поменялась только бизнес-составляющая (в лекции
сортировка строк, у нас — расчёт цены).

#### Observer (поведенческий)

> «Observer — поведенческий паттерн, который создаёт механизм
> подписки, позволяющий одним объектам следить и реагировать на
> события, происходящие в других объектах».
> *(TypeScript+2.pdf, слайд «Observer»)*

`Newsletter` хранит массив `Subscriber[]` и оповещает всех методом
`publish`. Подписчики реализуют интерфейс `Subscriber` и метод
`receive`. Конкретные подписчики — `EmailReader` и `SmsReader` —
по-разному отображают пришедшее сообщение. Структура повторяет схему
из лекции (`Subject` + `Observer`), просто в качестве события у нас
текст рассылки, а не случайное число.
